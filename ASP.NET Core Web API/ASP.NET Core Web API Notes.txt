Web API Development [RESTFul Architecture]
	Representational State Transfer

Rules to follow for implementing REST APIs
	1) The resources will be accessed using Only URIs
	2) Works with HTTP Protocol
	3) They don't have built-in encryption
	4) Perform CRUD operations using HTTP Nouns/Methods [GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS]
	5) It should not persist the state during request and response lifecycle [No Session]
	6) No Views. It only works with the data
	7) Cacheable 
	8) Supports Versioning 

ASP.NET Core Web API support - 
	1) Controller based development [Monolithic Development]
	2) Minimal API Development [Microservices Development] [Method/Function]

HTTP Protocol - 
	GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS


Head - is similar to get method except it does not return body like get. It is used for verification purpose like does resource exist, preflight requests, etc.

Options - Used for checking CORS and Http methods like GET/POST/PUT/DELETE ...
